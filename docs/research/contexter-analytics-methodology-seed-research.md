## Signals

### Signal 1: North Star Metric (NSM) — single value-expressing metric
- **Finding:** NSM is "the single metric that best captures the core value that your product delivers to customers" (Sean Ellis, 2010, inspired by Polaris). Amplitude's 2017 North Star Playbook formalized it as: (a) expresses value, (b) represents vision, (c) is a leading indicator, (d) actionable, (e) understandable, (f) measurable, (g) not vanity. NSM sits atop a small set of input metrics ("contributing inputs") that teams can influence directly.
- **Sources:** [Amplitude North Star Playbook](https://amplitude.com/books/north-star/why-use-the-north-star-framework) (2017), [Open Practice Library](https://openpracticelibrary.com/practice/north-star-framework/) (2024), [PostHog North Star guide](https://posthog.com/founders/north-star-metrics)
- **Confidence:** HIGH
- **Relevance to Contexter:** Most-cited framework for early-stage products. PostHog explicitly recommends it. Forces Contexter to name what value it delivers in a single number — likely something like "weekly documents ingested that produce successful searches" or "weekly MCP tool calls with useful result."
- **Tier:** FRAMEWORK

### Signal 2: AARRR / Pirate Metrics — funnel-based measurement
- **Finding:** Dave McClure's 2007 framework (Acquisition, Activation, Retention, Referral, Revenue) groups metrics by user-journey stage. Still relevant 18 years later per PostHog; common critique is you can't optimize all 5 simultaneously — pick one focus per stage. For B2B SaaS most practitioners prioritize Retention as PMF proxy, then expand.
- **Sources:** [PostHog AARRR explainer](https://posthog.com/product-engineers/aarrr-pirate-funnel), [Product Compass: AARRR](https://www.productcompass.pm/p/aarrr-pirate-metrics), [Inc magazine](https://www.inc.com/walter-chen/aarrr-dave-mcclure-s-pirate-metrics-and-the-only-five-numbers-that-matter.html)
- **Confidence:** HIGH
- **Relevance to Contexter:** Natural scaffold for 6 payment flows + MCP event taxonomy. Directly maps to Contexter funnel: Registration→Email verify→First doc→MCP connected→First search→Supporter payment→Subscription.
- **Tier:** FRAMEWORK

### Signal 3: HEART Framework — UX-layer metrics complement
- **Finding:** Rodden/Hutchinson/Fu's 2010 Google ACM CHI paper (Happiness, Engagement, Adoption, Retention, Task success). Designed to help designers focus amid data overwhelm. Uses Goals→Signals→Metrics (GSM) mapping for each dimension. Happiness = survey (NPS, ease of use); Task success = completion of key workflows.
- **Sources:** [Kerry Rodden's own site](https://kerryrodden.com/heart/), [Google Research paper (2010)](https://research.google/pubs/measuring-the-user-experience-on-a-large-scale-user-centered-metrics-for-web-applications/), [IxDF 2026 topic](https://ixdf.org/literature/topics/heart-framework)
- **Confidence:** HIGH
- **Relevance to Contexter:** Complements AARRR (which is funnel-focused) with UX quality. "Task success" = MCP tool call returning useful result; "Happiness" = supporter survey.
- **Tier:** FRAMEWORK

### Signal 4: Lean Analytics OMTM + Five Stages
- **Finding:** Croll/Yoskovitz 2013 book proposes "One Metric That Matters" per stage. Five stages: Empathy (problem-solution fit) → Stickiness (retention) → Virality (growth) → Revenue → Scale. At each stage, the OMTM is different — obsessing over revenue during Empathy/Stickiness is malpractice.
- **Sources:** [Lean Analytics Book: OMTM page](https://leananalyticsbook.com/one-metric-that-matters/), [Ash Maurya Medium](https://medium.com/lean-stack/lean-analytics-the-one-metric-that-matters-and-other-provocations-fd3006aab17), [Holistics Lean Analytics Pt 1](https://www.holistics.io/blog/lean-analytics-part-1-an-introduction-to-analytical-thinking/)
- **Confidence:** HIGH
- **Relevance to Contexter:** Contexter is pre-launch with 0 users = Empathy stage. OMTM for Empathy is "does the problem exist?" — not MRR. Signals suggest focusing on interview-derived success rate and waitlist-to-first-doc conversion, not monthly revenue until Stickiness stage (~100 supporters).
- **Tier:** FRAMEWORK

### Signal 5: Sean Ellis 40% PMF Test
- **Finding:** One-question survey: "How would you feel if you could no longer use [product]?" Score = % answering "very disappointed." Benchmark: ~40% = strong PMF, sustainable growth possible. Superhuman went 22% → 58% using this as an engine (5-step framework: survey → segment High Expectation Customers → analyze → iterate → re-measure).
- **Sources:** [First Round Review on Superhuman](https://review.firstround.com/how-superhuman-built-an-engine-to-find-product-market-fit/), [PMF Survey site](https://pmfsurvey.com/), [Reforge brief](https://www.reforge.com/blog/brief-how-superhuman-s-ceo-reverse-engineered-product-market-fit)
- **Confidence:** HIGH
- **Relevance to Contexter:** Primary PMF-measurement tool. Should be deployed after 100 supporters sign up. HXC segmentation may reveal "non-technical knowledge worker" vs "dev with Claude Desktop" as separate products.
- **Tier:** FRAMEWORK

### Signal 6: Build-Measure-Learn Loop (Lean Startup)
- **Finding:** Eric Ries's 2011 BML loop: Build MVP → Measure results → Learn → iterate. Each cycle starts with an explicit hypothesis (testable assumption). Core discipline: select actionable metrics tied to hypothesis, ignore vanity metrics.
- **Sources:** [Amplitude BML guide](https://amplitude.com/blog/build-measure-learn-the-product-management-lifecycle-loop), [ProdPad glossary](https://www.prodpad.com/glossary/build-measure-learn/), [Sorenkaplan template](https://www.sorenkaplan.com/build-measure-learn-loop-template/)
- **Confidence:** HIGH
- **Relevance to Contexter:** Methodology for connecting product decisions to metrics. Example hypothesis for Contexter: "We believe non-technical users will adopt if MCP config copy reduces setup to <2 min. If we measure 60%+ first-search-within-24h after config copy, we persevere."
- **Tier:** FRAMEWORK

### Signal 7: GQM (Goals-Questions-Metrics) — top-down metric design
- **Finding:** Basili/Weiss/Rombach 1980s framework from NASA SEL. Three levels: Goal (conceptual "what we want") → Questions (operational "what characterizes achievement") → Metrics (quantitative answers). Enforces top-down: every metric must trace to a question, every question to a goal.
- **Sources:** [UMD Basili canonical paper](https://www.cs.umd.edu/users/mvz/handouts/gqm.pdf), [Wikipedia GQM](https://en.wikipedia.org/wiki/GQM), [Flevy GQM guide](https://flevy.com/blog/goal-question-metric-gqm-framework/)
- **Confidence:** HIGH
- **Relevance to Contexter:** Alternative to AARRR if nopoint wants to start from goals (not funnel stages). Good discipline for CTX-11 spec: each tracked event must answer a specific question that serves a named goal. Prevents event sprawl.
- **Tier:** FRAMEWORK

### Signal 8: Jobs-to-be-Done (JTBD) Customer Value Metrics
- **Finding:** Christensen popularized (2003), Ulwick formalized via Outcome-Driven Innovation (ODI). CVMs = customer-defined success criteria for getting a job done. Products winning markets perform the job 20%+ better on these CVMs. ODI claims 86% success rate vs 17% industry average.
- **Sources:** [Strategyn: JTBD origin](https://strategyn.com/jobs-to-be-done/), [Christensen Institute theory](https://www.christenseninstitute.org/theory/jobs-to-be-done/), [GoPractice JTBD](https://gopractice.io/product/jobs-to-be-done-the-theory-and-the-frameworks/)
- **Confidence:** HIGH
- **Relevance to Contexter:** Non-obvious angle: instead of tracking events, first define "jobs" Contexter serves (e.g., "When I'm researching, help me recall my own notes inside Claude"). CVMs for that job = time-to-retrieval, answer accuracy, MCP tool setup time. This is what to actually measure.
- **Tier:** FRAMEWORK

### Signal 9: OKRs vs KPIs — different purposes
- **Finding:** OKRs = where we want to be (goal-driven, collaborative, forward-looking, quarterly). KPIs = where we are (ongoing benchmark targets). They complement: OKRs set direction, KPIs monitor steady-state. Using only KPIs produces status quo; using only OKRs produces no operational baseline.
- **Sources:** [ClearPoint OKR vs KPI](https://www.clearpointstrategy.com/blog/okrs-vs-kpis), [What Matters (Doerr)](https://www.whatmatters.com/resources/difference-between-okr-kpi), [Asana 2025 update](https://asana.com/resources/okr-vs-kpi)
- **Confidence:** HIGH
- **Relevance to Contexter:** Solo founder at pre-launch doesn't need formal OKRs (too much ceremony). But concept is useful: nopoint's "first 100 supporters × $10" = OKR (ambitious goal + measurable KR); MRR, churn, DAU = KPIs (ongoing monitoring).
- **Tier:** FRAMEWORK

### Signal 10: PLG (Product-Led Growth) metrics
- **Finding:** 58% of companies in 2026 use PLG. Canonical metrics: activation rate (20-40% normal for PLG), time-to-value (<10 min = elite), PQLs (product-qualified leads = behavioral thresholds signaling intent). Developer tools especially well-suited to PLG.
- **Sources:** [ProductLed metrics](https://www.productled.org/foundations/product-led-growth-metrics), [Mixpanel PLG 2026](https://mixpanel.com/blog/product-led-growth/), [Contentsquare PLG metrics](https://contentsquare.com/guides/product-led-growth/metrics/)
- **Confidence:** HIGH
- **Relevance to Contexter:** Contexter is classic PLG product (API-first, MCP). PQL concept useful: "user who uploaded ≥3 docs AND connected MCP AND ran ≥5 searches in week 1" is a PQL ready for supporter upgrade prompt.
- **Tier:** FRAMEWORK

### Signal 11: Vanity vs Actionable Metrics (Ries)
- **Finding:** Eric Ries, Lean Startup. Vanity metrics (page views, total signups, downloads) feel good but don't tie action→outcome. Actionable metrics satisfy 3As: Actionable (link change to result), Accessible (non-technical can understand), Auditable (traceable). Conversion rate = actionable; total registered users = vanity.
- **Sources:** [Eric Ries on Tim Ferriss blog (orig 2009)](https://tim.blog/2009/05/19/vanity-metrics-vs-actionable-metrics/), [Boldare guide](https://www.boldare.com/blog/lean-startup-vanity-metrics-vs-actionable-metrics/), [LeanStack 3 rules](https://blog.leanstack.com/3-rules-to-actionable-metrics-in-a-lean-startup/)
- **Confidence:** HIGH
- **Relevance to Contexter:** Specific traps for Contexter: "total documents uploaded" (vanity) vs "% of users who uploaded doc AND searched within 24h" (actionable); "Hacker News upvotes" (vanity) vs "HN→registration→first-search conversion" (actionable).
- **Tier:** ANTI-PATTERN

### Signal 12: Goodhart's Law — measure-becomes-target
- **Finding:** "When a measure becomes a target, it ceases to be a good measure" (Charles Goodhart, 1975). Software examples: story point inflation, code-coverage gaming, signup count optimization leading to email spam tactics. Historical: cobra bounty, nail-count vs nail-weight. Jellyfish, Axify document engineering-metric gaming.
- **Sources:** [Wikipedia Goodhart's Law](https://en.wikipedia.org/wiki/Goodhart's_law), [Jellyfish SE perspective](https://jellyfish.co/blog/goodharts-law-in-software-engineering-and-how-to-avoid-gaming-your-metrics/), [Axify analysis](https://axify.io/blog/goodhart-law), [PMC biomedical](https://pmc.ncbi.nlm.nih.gov/articles/PMC7901608/)
- **Confidence:** HIGH
- **Relevance to Contexter:** Direct risk if single metric is held up as north star without guardrails. E.g., "supporter signups" as target could incentivize giving away supporter tier for weak reasons, burning the 100-slot program.
- **Tier:** ANTI-PATTERN

### Signal 13: Dashboard / Data Graveyard Anti-Pattern
- **Finding:** Dashboards exist but no one looks at them — common in product orgs. Reasons: (a) misalignment with actual decision moments (decisions happen in Slack/calls/tickets, not dashboards), (b) information overload, (c) false sense of measurement. Specific case: B2B SaaS built 34-metric exec dashboard; executives spent 15 min scanning, couldn't identify priorities, stopped using it.
- **Sources:** [Bagel.ai: Dashboard Graveyard](https://bagel.ai/blog/the-dashboard-graveyard-how-product-teams-actually-use-insight-today/), [Oracle Analytics blog](https://blogs.oracle.com/analytics/dashboards-dead-or-alive-the-evolution-from-data-graveyards-into-data-gold-mines), [MotherDuck vibecoding dashboards](https://motherduck.com/blog/vibecoding-dashboards-best-practices/)
- **Confidence:** HIGH
- **Relevance to Contexter:** Highest-priority warning for solo founder. Solution: metrics must live where decisions happen (weekly email digest, Slack alert on threshold, not web dashboard nopoint must visit).
- **Tier:** ANTI-PATTERN

### Signal 14: Survivorship Bias + Simpson's Paradox in Cohorts
- **Finding:** Cohort analysis focused only on "surviving" cohorts hides churn drivers; aggregated data can reverse direction when disaggregated by segment (Simpson's). Mixpanel example: technique "beneficial within each user group" shows "negative in aggregate" due to uneven group distribution.
- **Sources:** [Mixpanel on Simpson's Paradox](https://mixpanel.com/blog/avoiding-data-fallacies-and-biases-simpsons-paradox-and-the-importance-of-segmenting-data/), [Wikipedia Survivorship](https://en.wikipedia.org/wiki/Survivorship_bias), [Data Analysis Guide book](https://bookdown.org/mike/data_analysis/sec-survivorship-bias.html)
- **Confidence:** HIGH
- **Relevance to Contexter:** Specific risk: comparing "dev with Claude Desktop" vs "non-tech user" cohorts in aggregate could mislead. Always segment first (supporter tier, MCP client type, acquisition channel).
- **Tier:** ANTI-PATTERN

### Signal 15: Goodhart + Premature Optimization in pre-PMF
- **Finding:** Multiple sources warn: early-stage companies should not obsess over LTV/CAC or Rule of 40 — those metrics are stable only post-PMF. Pre-PMF focus should be revenue validation + activation, not ratios. Wall Street Prep and HBS concur: early-stage LTV:CAC naturally <1.
- **Sources:** [Wall Street Prep LTV/CAC](https://www.wallstreetprep.com/knowledge/ltv-cac-ratio/), [HBS Online](https://online.hbs.edu/blog/post/ltv-cac), [SaaS Capital 2025 benchmarks](https://www.saas-capital.com/blog-posts/benchmarking-metrics-for-bootstrapped-saas-companies/)
- **Confidence:** MEDIUM
- **Relevance to Contexter:** Contexter should NOT implement LTV/CAC dashboards in CTX-11. Too early. Focus on activation + retention + revenue validation.
- **Tier:** ANTI-PATTERN

### Signal 16: Guardrail Metrics concept
- **Finding:** Counter-metrics monitored alongside primary metrics to catch "bad wins" — changes that improve X while silently degrading Y. Airbnb runs thousands of experiments/month; guardrails trigger ~25 reviews/month, ~5 pauses. Classic example: simplified signup improves trial count but "conversion to paid" guardrail stops if it drops.
- **Sources:** [PostHog guardrail metrics](https://posthog.com/product-engineers/guardrail-metrics), [Mixpanel guardrails](https://mixpanel.com/blog/guardrail-metrics/), [Statsig guardrails](https://www.statsig.com/blog/what-are-guardrail-metrics-in-ab-tests)
- **Confidence:** HIGH
- **Relevance to Contexter:** Direct applicability. NSM + 3-5 guardrails. E.g., NSM = "weekly active searches"; guardrails = abuse rate, MCP error rate, supporter-cancellation rate. Prevents single-metric gaming.
- **Tier:** FRAMEWORK

### Signal 17: Aha Moment + Activation Event methodology
- **Finding:** Aha moment = subjective "user realized value"; activation event = measurable action correlating to long-term retention. Canonical examples: Facebook "7 friends in 10 days"; Slack "2,000 team messages"; Twitter "30 follows." Find it by running retention analysis on different first-week behaviors and picking the one most correlated with Week 4+ retention.
- **Sources:** [Appcues aha moment guide](https://www.appcues.com/blog/aha-moment-guide), [Chameleon onboarding](https://www.chameleon.io/blog/successful-user-onboarding), [Encharge 17 examples](https://encharge.io/aha-moment-examples/), [Kissmetrics activation](https://www.kissmetrics.io/blog/activation-rate-optimization)
- **Capabilities:** Detectable even pre-PMF after ~50-100 users.
- **Confidence:** HIGH
- **Relevance to Contexter:** Mission-critical to define. Hypothesis candidates: "User uploads 3 docs AND connects MCP AND runs 5 searches in week 1" or "User completes first ask_about_document successfully." Must be discovered empirically, not guessed.
- **Tier:** FRAMEWORK

### Signal 18: Retention Curves (Smile / Flat / Decline) — a16z canonical
- **Finding:** Three canonical shapes. (1) Declining = all users eventually churn (unsustainable); (2) Flat = core user group retains indefinitely (sustainable business); (3) Smile = churned users return over time (exceptional — Slack, Facebook). Flat curve is PMF signal; smile curve is category leader signal.
- **Sources:** [Sequoia Retention article](https://articles.sequoiacap.com/retention), [Churnkey retention curves](https://churnkey.co/blog/retention-curves/), [a16z growth podcast](https://a16z.com/podcast/a16z-podcast-the-basics-of-growth-engagement-retention/), [Indie Hackers: does your cohort smile?](https://www.indiehackers.com/post/does-your-cohort-retention-curve-smile-8c574ccdcb)
- **Confidence:** HIGH
- **Relevance to Contexter:** Required analysis once 100+ supporters. If retention curve trends to 0, no amount of acquisition saves it. Flat curve by month 3 = proceed with confidence.
- **Tier:** FRAMEWORK

### Signal 19: DAU/WAU/MAU + Stickiness Ratio
- **Finding:** Stickiness = DAU/MAU. Benchmarks: >20% good, >25% excellent, social apps 50%+. Gaming 20-30%, fintech 10-22%, e-commerce ~10%. L7 (days active in last 7) and L28 (days active in last 28) are distribution measures more granular than DAU/MAU. Choose "active" definition based on meaningful engagement (not just login).
- **Sources:** [Sequoia selecting user metric](https://medium.com/sequoia-capital/selecting-the-right-user-metric-de95015aa38), [Mixpanel on MAU/WAU/DAU](https://mixpanel.com/blog/mau/), [Clevertap stickiness](https://clevertap.com/blog/dau-vs-mau-app-stickiness-metrics/)
- **Confidence:** HIGH
- **Relevance to Contexter:** For MCP-centric product, "active" = "completed ≥1 MCP tool call with successful response." DAU/MAU 10-20% realistic for research-tool use case (not daily-use messenger).
- **Tier:** BENCHMARK

### Signal 20: RAG-Specific Quality Metrics (Ragas / DeepEval)
- **Finding:** 5 canonical RAG eval metrics: Answer Relevancy, Faithfulness, Contextual Relevancy, Contextual Recall, Contextual Precision. Tools: Ragas, DeepEval, Braintrust. Modern practice: CI/CD gates on RAG quality, not just deploy-and-pray. Critical: map business objectives (time-to-insight, user satisfaction) to technical metrics rather than default to BLEU/recall.
- **Sources:** [Confident AI RAG metrics](https://www.confident-ai.com/blog/rag-evaluation-metrics-answer-relevancy-faithfulness-and-more), [Ragas docs](https://docs.ragas.io/en/stable/concepts/metrics/available_metrics/), [Qdrant RAG eval](https://qdrant.tech/blog/rag-evaluation-guide/), [Atlan RAG eval 2026](https://atlan.com/know/how-to-evaluate-rag-systems-explained/), [Evidently AI guide](https://www.evidentlyai.com/llm-guide/rag-evaluation)
- **Confidence:** HIGH
- **Relevance to Contexter:** Contexter is a RAG product — these are the product-quality metrics, separate from business/engagement metrics. Should be in separate dashboard from AARRR funnel. Hallucination rate (inverse of faithfulness) = user-trust KPI.
- **Tier:** FRAMEWORK

### Signal 21: MCP-Specific Observability Patterns (emerging 2025)
- **Finding:** 2025 established OpenTelemetry GenAI semantic conventions for MCP. Datadog's MCP server emits two standard metrics: `datadog.mcp.session.starts` (per session init), `datadog.mcp.tool.calls` (per tool call, tagged with tool_name). Sentry offers protocol-aware MCP monitoring (transport, tool, client breakdown). MCP servers widely considered "black boxes" without explicit instrumentation.
- **Sources:** [SigNoz MCP observability](https://signoz.io/blog/mcp-observability-with-otel/), [MintMCP OTel for MCP](https://www.mintmcp.com/blog/opentelemetry-ai-agents), [Glama OTel for MCP](https://glama.ai/blog/2025-11-29-open-telemetry-for-model-context-protocol-mcp-analytics-and-agent-observability), [Sentry MCP monitoring](https://blog.sentry.io/introducing-mcp-server-monitoring/), [Datadog MCP server](https://www.datadoghq.com/product/ai/mcp-server/)
- **Confidence:** HIGH
- **Relevance to Contexter:** Direct implementation guidance for Contexter MCP taxonomy. `tool_name` tag + session correlation = minimum viable instrumentation. Likely cheaper to use OTel + PostHog than Datadog, but naming/tagging conventions should align.
- **Tier:** TOOL-PATTERN

### Signal 22: Dogfooding as analytics discipline
- **Finding:** PostHog, Linear, OpenAI explicitly dogfood their analytics and product tools. Critical insight: PM using own product daily feels friction that analytics might not surface. Generates automatic dogfood-dataset (your own team's usage) that complements real-user data. Fast feedback loop between building and experiencing.

- **Confidence:** HIGH
- **Relevance to Contexter:** nopoint already uses Contexter (Artem knowledge hub). Should explicitly track own usage as zero-cost first dataset. But: single-user dogfooding is not generalizable — need real user data to avoid confirmation bias.
- **Tier:** PATTERN

### Signal 23: Leading vs Lagging Indicators
- **Finding:** Leading = predictive (trial signups, demo calls, activation, DAU); Lagging = past outcome (revenue, churn, retention). Leading gives founders "room to react"; lagging confirms outcome. Best practice: pair them — activation rate (leading) + 90-day retention (lagging).
- **Sources:** [Amplitude leading vs lagging](https://amplitude.com/blog/leading-lagging-indicators), [Mercury blog](https://mercury.com/blog/leading-versus-lagging-indicators), [Waveup](https://waveup.com/blog/leading-vs-lagging-metrics/), [Whatfix 20 examples](https://whatfix.com/blog/leading-vs-lagging-indicators/)
- **Confidence:** HIGH
- **Relevance to Contexter:** Event taxonomy should explicitly tag which events produce leading vs lagging metrics. First 100 supporters = lagging; weekly activation rate = leading for it.
- **Tier:** FRAMEWORK

### Signal 24: Input vs Output metrics (Grove) + Paired Indicators
- **Finding:** Andy Grove's High Output Management (1983): "Measure breakfasts sold, not eggs boiled." Output = impact; activity/input = work expended. Complementary principle: pair indicators so effect+counter-effect are measured (e.g., sales + quality; speed + error rate). Output-only metric can incentivize quality degradation without the paired counter.
- **Sources:** [Medium: High Output PM](https://medium.com/pm-insights/high-output-product-management-335adbc84874), [Because Tech Rocks](https://www.becausetech.rocks/blog/high-output-management/), [Wikipedia](https://en.wikipedia.org/wiki/High_Output_Management)
- **Confidence:** HIGH
- **Relevance to Contexter:** Core discipline. Pairs for Contexter: MCP tool calls (output) + error rate (counter); supporter signups (output) + satisfaction score (counter); documents uploaded (output) + retrieval success rate (counter).
- **Tier:** FRAMEWORK

## Pattern Observations

### Pattern A: Single-metric anchor + contributing inputs (North Star pattern)
- One top metric expresses value, 3-6 input metrics influence it. Used by Amplitude (playbook), PostHog, Airbnb, Spotify.
- Strength: forces clarity on "what does this product do for the user?"
- Weakness: single metric = Goodhart risk; needs guardrails.

### Pattern B: Stage-gated OMTM (Lean Analytics pattern)
- Different single metric per business stage (Empathy → Stickiness → Virality → Revenue → Scale).
- Forces founders to ignore revenue metrics until product works.
- Used by Ash Maurya, Croll/Yoskovitz, startup accelerators.

### Pattern C: Funnel-stage categorization (AARRR / PLG pattern)
- Group metrics by user journey stage: Acquisition / Activation / Retention / Referral / Revenue.
- Works well when decisions differ per stage (marketing vs onboarding vs product).
- Used by PostHog, Amplitude (pre-NSM), Reforge (with Loops refinement).

### Pattern D: Goal-traceability (GQM pattern)
- Every metric traces up to a question that serves a goal. Prevents event sprawl.
- Used by software engineering metrics community, NASA SEL.
- Under-represented in consumer / SaaS but ideal for disciplined solo founder.

### Pattern E: Hypothesis-driven iteration (BML pattern)
- Build MVP to test hypothesis → measure → learn → iterate. Metric exists only in service of hypothesis.
- Used by Lean Startup community, most modern product teams.
- Compatible with all other patterns (provides the "why" for metrics).

### Pattern F: Open Startup transparency
- Public dashboards of MRR/ARR/customers/churn. Buffer, Pieter Levels, Plausible, Bannerbear, Baremetrics Open Startups, Senja.
- Converts measurement into marketing. Attracts early adopters and followers.
- Weakness: exposes weakness during stagnation; requires emotional readiness for public failure.

### Pattern G: Dogfood + empirical + external
- Founder's own usage + internal team + real users, as three complementary datasets.
- Used by PostHog, Linear, OpenAI. Avoids confirmation bias of dogfood-only.

### Pattern H: Ambient metric delivery (weekly digest, Slack alert)
- Push metrics to where decisions happen (Slack, email, CLI), not where dashboards live (web apps).
- Emerges from dashboard graveyard research. Not branded yet as a "framework" but consistent practice among disciplined indie founders.

### Pattern I: NSM + guardrails + paired indicators
- Primary NSM, 2-4 guardrail metrics to catch "bad wins," plus Grove's paired-indicator principle (output + quality counter).
- Used in A/B testing at Airbnb, Microsoft, Google; transferable to product metrics broadly.

### Pattern J: Business-metric-to-tech-metric mapping (RAG-specific)
- Map business objective (user satisfaction, time-to-insight, cost-per-query) to technical metric (recall@k, faithfulness, latency).
- Emerging 2025 practice, specific to AI-first products.
- Novel for Contexter: most SaaS playbooks don't cover RAG-quality as separate metric layer.

## Noise (captured but deprioritized)

1. **Full OKR system with quarterly cadence** — overkill for solo founder pre-launch; too much ritual for a single person. Use OKR concept (ambitious goal + measurable KRs) informally, not as weekly ceremony. Revisit post-$10K MRR.
2. **Rule of 40** — applies only post-PMF companies at $1M+ ARR with consistent growth rate. Irrelevant at 0 users. Revisit at $500K ARR.
3. **LTV/CAC ratio obsession** — naturally <1 for bootstrapped SaaS in first year (per Wall Street Prep, HBS). Measure but don't optimize; pre-PMF focus is activation + retention. Revisit post-100 supporters.
4. **NDCG / MRR (Mean Reciprocal Rank)** — classical IR metrics for tuning retrieval. Useful for Contexter engineering but too technical for business-level analytics methodology. Save for RAG-eval DEEP.
5. **MQL/SQL pipeline** — marketing-sales pipeline concept from enterprise SaaS. Irrelevant at solo + 0 sales team. Skip until $30K MRR.
6. **Multi-touch U-shape / J-shape attribution** — overkill at <100 conversions/month. First-touch + last-touch is enough signal.
7. **GAMES (Google post-HEART)** — insufficient primary-source coverage; HEART remains the canonical Google framework. Low priority unless specific evidence emerges.
8. **Superhuman PMF Engine full 5-step machinery** — design for Superhuman's $30/mo business with sales team. Contexter can use the 40% test without Vohra's full machinery. Extract core (survey + HXC segmentation), skip 5-step framework.
9. **Complex growth loop modeling** — Balfour's loops require signal volume to observe. At <100 users they're hypothesis, not measured. Defer until Stickiness stage.
10. **Customer Success / NPS / CSAT formal programs** — require headcount. Solo founder proxies via direct supporter conversations + 40% test.
11. **Full stack of RAG eval tools (Ragas, DeepEval, Braintrust all installed)** — pick one. Ragas is de facto standard for Python + LangChain ecosystems.
12. **BigCo metric dashboards (Airbnb guardrail machinery, 34-metric executive dashboards)** — explicitly validated as anti-pattern. Solo founder needs ≤10 metrics total in view.

## Unexpected findings (researcher freedom deviations)

1. **Pattern H (Ambient metric delivery) emerged as strongest anti-graveyard solution** but is not a named framework in the literature. Consistent practice across indie founders (push-to-Slack, weekly digest) should become a named pattern. Elevated to DEEP-5 Layer 3.

2. **Grove's paired-indicator principle** (High Output Management, 1983) is the underrated complement to Goodhart's Law. Most Goodhart discussions say "don't use single metrics as targets" — Grove gives the *constructive* solution: always pair with a counter-metric. This is essentially the ancestry of guardrail metrics in modern A/B testing (Airbnb, Microsoft). Worth explicit call-out in Contexter methodology.

3. **MCP observability is nascent** but converging on OpenTelemetry GenAI conventions + Datadog/Sentry protocol-aware approach (2025). The `mcp.session.starts` and `mcp.tool.calls` naming is becoming de facto. Contexter should align to this convention even if using PostHog instead of Datadog — cheap optionality for future migration.

4. **Indie Hackers 17x better conversion than Product Hunt** (OpenHunts 2024 study) is a non-obvious finding. If validated, it overturns the typical "launch on PH then IH" playbook. The 4-6 months sustained-IH-participation requirement is the cost. Relevant to Contexter's channel strategy independent of analytics methodology.

5. **Lean Analytics 5 stages** (Empathy → Stickiness → Virality → Revenue → Scale) are rarely quoted but provide the cleanest answer to "what OMTM at what stage?" Contexter is in Empathy. Revenue OMTM is stage-wrong. This deserves explicit framing in L3 spec: we're NOT chasing MRR yet, we're chasing empathy signals.

6. **Plausible's "5-10 metrics vs Google's 300"** is a measurable discipline data point. If Marko Saric (co-founder of a $1M+ ARR analytics company) uses 5-10 metrics, solo-founder benchmark should probably be ≤10 in active view.

7. **JTBD Customer Value Metrics** as a measurement layer (not just discovery tool) — under-represented in mainstream analytics writing, but powerful for Contexter: define "the job" then define what metrics the *user* uses to know the job is done well.

8. **"Decision problem, not data problem"** (Bagel.ai framing) — elegant reframing of the dashboard graveyard. nopoint's CTX-11 should explicitly state "this analytics system exists to support specific recurring decisions" and enumerate those decisions, not just dashboards.
