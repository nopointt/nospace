### Tauri/SolidStart bridge pattern

Uses **tauri-specta** for typesafe IPC:
- Rust side: `#[tauri::command] #[specta::specta] fn greet(...)`
- Build step auto-generates `src/bindings.ts` with fully typed `commands` + `events`
- Frontend: `import { commands, events } from "~/bindings"` — no raw invoke strings

```typescript
// bindings.ts (auto-generated)
export const commands = {
  async greet(name: string): Promise<string> {
    return await TAURI_INVOKE("greet", { name });
  },
};
export const events = __makeEvents__<{ demoEvent: DemoEvent }>({ demoEvent: "demo-event" });
```

### Key gotcha

**Port mismatch:** template `tauri.conf.json` has `devUrl: "http://localhost:3000"` but `app.config.ts` locks port to `1420`. In practice, `vinxi dev` starts on `1420` and Tauri reads the dev URL from `app.config.ts`, not `tauri.conf.json`. Verify after scaffolding.

**Static preset vs CF Pages preset:** Quantum uses `preset: "static"`. Harkly needs `preset: "cloudflare-pages"` for the web build. For Tauri, Harkly needs a separate build target or conditional config. These are incompatible in a single `app.config.ts` — solution: use env flag (`TAURI_ENV_PLATFORM`) to switch presets at build time.

### D1 + Drizzle setup — the canonical pattern

**drizzle.config.ts:**
```typescript
export default {
  schema: "./src/db/schema.ts",
  out: "./drizzle",
  dialect: "sqlite",
  dbCredentials: {
    // points to local wrangler miniflare state
    url: "./.wrangler/state/v3/d1/miniflare-D1DatabaseObject/{hash}.sqlite"
  }
} satisfies Config;
```

**src/db/schema.ts:**
```typescript
import { sqliteTable, text, customType } from "drizzle-orm/sqlite-core";
// Custom UUID type stored as blob (D1 supports Uint8Array blobs)
const uuid = customType<{ data: string; driverData: Uint8Array }>({...});
export const visitor = sqliteTable("visitor", {
  id: uuid().primaryKey().$defaultFn(() => uuidv7()),
  name: text().notNull()
});
```

**src/db/client.ts — key: DB comes from event.locals.env, not module-level:**
```typescript
import { drizzle } from "drizzle-orm/d1";
export const createDb = (event: FetchEvent): Database =>
  drizzle(event.locals.env.DB, { schema });
```

This is the critical D1 pattern: D1 binding is only available per-request. No module-level `drizzle()` call. Always pass through event.locals.

**Migration workflow:**
```sh
# Generate SQL migrations from schema changes
pnpm db:generate        # drizzle-kit generate

# Apply to local wrangler state
pnpm db:migrate         # wrangler d1 migrations apply solid-pages-preview --local

# Apply to remote D1 (production)
wrangler d1 migrations apply solid-pages-production
```

### Middleware pattern — how CF bindings reach SolidStart

```typescript
// src/middleware.ts
const cloudflare = async (event: FetchEvent): Promise<void> => {
  if (import.meta.env.DEV) {
    // Dev: use wrangler PlatformProxy to simulate CF environment
    const proxy = await wrangler.getPlatformProxy<Env>({ persist: true });
    event.locals.env = proxy.env;
    event.locals.cf = proxy.cf;
    event.locals.caches = proxy.caches;
  } else {
    // Prod: bindings come from CF runtime context
    const context = event.nativeEvent.context;
    event.locals.env = context.cloudflare.env;
    event.locals.cf = context.cf;
  }
};

const db = (event: FetchEvent): void => {
  event.locals.db = createDb(event);
};

export default createMiddleware({
  onRequest: [cloudflare, redirectToDomain, sentry, user, db]
});
```

The DEV branch uses `wrangler.getPlatformProxy()` to make D1/KV/R2 available during `vinxi dev`. This is the correct local dev pattern.

### R2 integration

R2 is accessed directly via `event.locals.env.R2`:
```typescript
await env.R2.put(r2Key, "Hello, World!");
const r2 = await env.R2.get(r2Key);
```

No presigned URLs shown — this is server-side direct access. For client-side uploads, the pattern would be: route handler generates presigned URL via Workers SDK, client uploads directly to R2.

### Auth

Uses Cloudflare Access JWT validation (not better-auth). JWT issuer/audience from env vars, validated in middleware. This is different from what Harkly needs (better-auth) but the middleware integration pattern is identical.

### TypeScript worker types

Generated by `wrangler types src/worker-configuration.d.ts`:
```typescript
declare namespace Cloudflare {
  interface Env {
    KV: KVNamespace;
    R2: R2Bucket;
    DB: D1Database;
    AI: Ai;
    ENVIRONMENT: "preview" | "production";
    // ...
  }
}
interface Env extends Cloudflare.Env {}
```

This gives full type safety for all CF bindings in `event.locals.env`.

### Key gotcha

## 3. solid-flow (canvas)

**Repo:** `miguelsalesvieira/solid-flow`
**Version:** `0.0.0-development` — pre-release, never reached 1.0
**Last meaningful activity:** 2022-2023 (targets solid-js ^1.5.1, rollup ^2.79.1)
**npm:** published as `solid-flow`

### What it does

A node graph editor for SolidJS — nodes are draggable, edges are SVG paths connecting node inputs/outputs. Inspired by React Flow.

### Architecture analysis

**Component hierarchy:**
```
SolidFlow (FlowChart)
  NodesBoard           ← div, handles mouse events, renders nodes
    NodeComponent      ← positioned via transform: translate(x,y)
  EdgesBoard           ← SVG overlay, renders bezier curves
    EdgeComponent      ← SVG path
```

**State management:** mixed createSignal + createStore. Nodes positions are in a signal array. Node data is a store. Edge positions are a signal object.

**Canvas implementation:** Not a true infinite canvas — uses CSS overflow:scroll on a fixed-size container:
```css
.wrapper { overflow: scroll; }
.content {
  height: 150vh;
  width: calc(1080px * 2);  /* fixed 2160px wide */
  background-size: 30px 30px;
  background-image: radial-gradient(circle, #b8b8b8bf 1px, rgba(0,0,0,0) 1px);
}
```

This is a **scrollable canvas**, not an infinite zoom+pan canvas. No transform matrix, no zoom, no pinch-to-zoom.

### Feature assessment

| Feature | Status |
|---------|--------|
| Node drag | Yes — mouse events, translate transform |
| Edge draw (connect nodes) | Yes — mousedown output → mouseup input |
| Edge delete | Yes |
| Node delete | Yes (opt-in via `actions.delete`) |
| Custom node content | Yes — JSX passed as `data.content` |
| Multiple inputs/outputs per node | Yes |
| Infinite canvas | NO — fixed scrollable area |
| Zoom | NO |
| Pan (drag background) | NO |
| Canvas transform matrix | NO |
| Touch/mobile | NO |
| Minimap | NO |
| Selection/marquee | NO |
| Undo/redo | NO |
| TypeScript strict | Partial — many `any` types |

### API surface

```typescript
<SolidFlow
  nodes={Node[]}      // {id, position: {x,y}, data: {label?, content}, inputs, outputs, actions?}
  edges={Edge[]}      // {id, sourceNode, targetNode, sourceOutput, targetInput}
  onNodesChange={(newNodes: Node[]) => void}
  onEdgesChange={(newEdges: Edge[]) => void}
/>
```

### Critical issues for Harkly

1. **No infinite canvas** — fixed 2160x150vh. For spatial knowledge visualization with many nodes, this is unusable.
2. **No zoom** — cannot zoom out to see graph overview.
3. **Abandoned development** — version pinned at `0.0.0-development`, no releases, issues open for years.
4. **solid-js 1.5.1** — current is 1.9.x. Likely works but not tested with modern SolidStart.
5. **No SSR guards** — `document.body.addEventListener` in `onMount` is fine but the library has no explicit SSR safety.
6. **State mutation** — uses `produce()` from solid-js/store but also direct array mutation patterns (`next[nodeIndex].x = ...` where `next = [...prev]`).

### Verdict

**Do not use solid-flow for Harkly.** It lacks the core requirements: infinite canvas, zoom, pan. It is effectively unmaintained. Using it would require rewriting the entire canvas layer from scratch anyway.

### What alternatives exist for SolidJS infinite canvas

See Section 5 (alternatives research) below.

## 4. SolidStart + better-auth integration

**Official docs:** https://www.better-auth.com/docs/integrations/solid-start
**Date checked:** 2026-03-19

### Route handler (required)

```typescript
// src/routes/api/auth/[...auth].ts
import { auth } from "~/lib/auth";
import { toSolidStartHandler } from "better-auth/solid-start";

export const { GET, POST } = toSolidStartHandler(auth);
```

Note: route must be at `/api/auth/[...auth]` (catch-all). The `[auth]` single-segment pattern (found in aguywhosaguy repo) also works but `[...auth]` is safer for all sub-paths.

### Auth instance (server-side) — the D1 problem

better-auth requires a database connection at init time. With D1, the binding is only available per-request (from `event.locals.env.DB`). This creates a tension:

**Option A: Lazy singleton with D1 binding (recommended for CF Workers)**
```typescript
// src/lib/auth.ts
import { betterAuth } from "better-auth";
import { drizzleAdapter } from "better-auth/adapters/drizzle";
import { drizzle } from "drizzle-orm/d1";

let authInstance: ReturnType<typeof betterAuth> | null = null;

export function getAuth(env: Env) {
  if (!authInstance) {
    const db = drizzle(env.DB, { schema });
    authInstance = betterAuth({
      database: drizzleAdapter(db, { provider: "sqlite" }),
      emailAndPassword: { enabled: true },
      // ...
    });
  }
  return authInstance;
}
```

Then in route handler:
```typescript
// src/routes/api/auth/[...auth].ts
import { toSolidStartHandler } from "better-auth/solid-start";
import { getRequestEvent } from "solid-js/web";

export const GET = async (event: Request) => {
  const requestEvent = getRequestEvent();
  const auth = getAuth(requestEvent!.locals.env);
  return toSolidStartHandler(auth)({ request: event } as any);
};
```

**Option B: Drizzle adapter with pg (Supabase/Postgres) — simpler**
```typescript
// src/lib/auth.ts — works if using Supabase, not D1
import { betterAuth } from "better-auth";
import { drizzleAdapter } from "better-auth/adapters/drizzle";
import db from "~/db/db"; // module-level drizzle(process.env.DATABASE_URL)

export const auth = betterAuth({
  database: drizzleAdapter(db, { provider: "pg" }),
  emailAndPassword: { enabled: true },
});
```

This is what `aguywhosaguy/aguywhosaguycom-solidstart` uses (Postgres, not D1).

### Client-side auth

```typescript
// src/lib/auth-client.ts
import { createAuthClient } from "better-auth/solid";
import { inferAdditionalFields } from "better-auth/client/plugins";
import { auth } from "./auth";

export const authClient = createAuthClient({
  baseURL: import.meta.env.VITE_APP_URL || "http://localhost:3000",
  plugins: [inferAdditionalFields<typeof auth>()]
});

export type Session = typeof authClient.$Infer.Session;
```

### Session access patterns

**Server-side (in server functions / route handlers):**
```typescript
import { auth } from "~/lib/auth";
const session = await auth.api.getSession({ headers: request.headers });
```

**Client-side (reactive):**
```typescript
import { authClient } from "~/lib/auth-client";
const session = authClient.useSession(); // returns Accessor
// access: session()?.data?.user (not session()?.user)
```

Note: better-auth wraps session in `.data` on the client. The correct path is `session()?.data?.user`, not `session()?.user`.

### Gotcha: D1 + better-auth schema migration

better-auth has its own CLI to generate schema:
```sh
npx @better-auth/cli generate --output src/db/auth-schema.ts
```

This outputs Drizzle schema for auth tables (user, session, account, verification). Then run `drizzle-kit generate` and `wrangler d1 migrations apply` as normal.

The `drizzle.config.ts` must point to the auth schema + app schema combined:
```typescript
export default {
  schema: ["./src/db/schema.ts", "./src/db/auth-schema.ts"],
  out: "./drizzle",
  dialect: "sqlite",
  dbCredentials: { url: "./.wrangler/state/v3/d1/..." }
};
```

## 5. Canvas alternatives research (solid-flow replacement)

Since solid-flow is unsuitable, evaluated alternatives:

### Option A: @xyflow/react ported manually (NOT recommended)

React Flow (xyflow) has no official SolidJS port. Porting is a major undertaking.

### Option B: Svelvet (Svelte-based)

SvelteJS. Not usable in SolidJS.

### Option C: Build custom infinite canvas

For Harkly's spatial canvas requirements, a custom implementation is likely necessary. The core primitives:

```
Container div (overflow: hidden, position: relative)
  Canvas div (transform: matrix(scale, 0, 0, scale, tx, ty))
    Nodes (position: absolute, left/top or translate)
  SVG overlay (same transform matrix)
    Edges (bezier paths)
```

Pan: drag on background → update tx, ty
Zoom: wheel event → update scale, adjust tx/ty to zoom toward cursor
Node drag: pointer events on individual nodes

This is ~500-800 lines of SolidJS to implement correctly with:
- Pointer event handling (not mouse — supports touch)
- Transform matrix math
- Edge routing (bezier control points from port positions)
- Virtualization (for 100+ nodes)

### Option D: Use @xyflow/xyflow (framework-agnostic core — NEW in 2024)

XYFlow team extracted the core into a framework-agnostic package. Check if SolidJS wrapper exists.

### Option E: Penpot's SolidJS canvas patterns

Penpot uses a custom canvas. Their SolidJS components may have useful patterns.

### Option F: Konva.js (canvas 2D)

Not SVG — renders to `<canvas>` element. Good performance for many nodes. No native SolidJS integration but wrapping is straightforward.

### Recommendation for canvas

Build a lightweight custom infinite canvas for Harkly. Key reasons:
1. Harkly has specific visual requirements (Bauhaus design system, spatial metaphor)
2. Generic libraries add 100KB+ of overhead
3. solid-flow demonstrates the architecture clearly — it's ~400 lines of SolidJS, the missing part is the transform matrix

Steal the SVG edge rendering pattern from solid-flow, replace the scroll-based canvas with a transform matrix approach.

### Build matrix

| Target | preset | ssr | Command |
|--------|--------|-----|---------|
| Cloudflare Pages (web) | `cloudflare-pages` | true | `vinxi build` → `wrangler pages deploy` |
| Tauri desktop | `static` | false | `tauri build` (calls `vinxi build` with TAURI env) |
| Local dev (web) | `cloudflare-pages` | true | `vinxi dev` (PlatformProxy for D1/KV/R2) |
| Local dev (desktop) | `static` | false | `tauri dev` |

**Critical:** `app.config.ts` needs to switch preset based on `process.env.TAURI_ENV_PLATFORM`:

```typescript
const isTauri = Boolean(process.env.TAURI_ENV_PLATFORM);

export default defineConfig({
  ssr: !isTauri,
  server: {
    preset: isTauri ? "static" : "cloudflare-pages",
    ...(isTauri ? {} : { compatibilityDate: "2024-11-03" })
  },
  vite: () => ({
    server: isTauri ? { port: 1420, strictPort: true } : {},
    envPrefix: ["VITE_", "TAURI_"],
  })
});
```

### Canvas strategy

Build a custom lightweight infinite canvas in SolidJS. Do NOT use solid-flow. The canvas needs:

1. **Transform matrix layer** — single `<div>` with `transform: matrix(scale,0,0,scale,tx,ty)` containing all nodes
2. **SVG edge overlay** — `<svg>` element with matching transform, bezier paths
3. **Pointer events** — `onPointerDown/Move/Up` on the container for pan; `onWheel` for zoom
4. **Node drag** — `onPointerDown` on individual nodes, capture pointer, update position
5. **Edge routing** — calculate control points from port DOM rects (like solid-flow's onMount pattern)

Estimated size: ~600-800 lines in 4-5 SolidJS files. Avoid virtualization in v1 (handle up to ~200 nodes without it).
