# CONTEXT ECONOMY REGULATION
> Applies to: all AI agents, all sessions, all projects under /nospace.
> Authority: nopoint (CEO). Review: per quarter.
> Tags: [context, tokens, economy, limits, efficiency, assistant]

---

## § 1 — Принцип

**Контекст — невозобновляемый ресурс сессии.** Каждый файл прочитанный без необходимости, каждый лишний шаг рассуждения — это токены которые нельзя вернуть. Агент ОБЯЗАН относиться к контексту как к бюджету, а не как к бесконечному пространству.

Ключевое правило: **делай минимум для цели**. Не читай то что не нужно прямо сейчас.

---

## § 2 — Оценка до начала работы

Перед выполнением любой операции агент ДОЛЖЕН мысленно оценить её контекстную стоимость:

```
Контекстная стоимость операции:
  LOW    — 1-3 файла, простое изменение, локальная задача
  MEDIUM — 4-10 файлов, несколько компонентов, анализ
  HIGH   — 10+ файлов, рефакторинг, исследование кодовой базы
  CRITICAL — чтение всего проекта, большой рефактор, migration
```

**Правило MEDIUM и выше:** агент обязан предупредить nopoint ДО начала:

> "⚠️ Эта операция потребует много контекста ([причина: сколько файлов, почему]). Предлагаю сделать иначе: [альтернатива 1] или [альтернатива 2]. Как поступим?"

Не начинать работу до подтверждения.

---

## § 3 — Сигналы дорогой операции

Агент ДОЛЖЕН поднимать флаг при виде любого из:

| Сигнал | Пример | Реакция |
|---|---|---|
| "прочитай весь проект" | "посмотри что есть в tLOS" | Предложить: grep по ключевым словам или читать только память |
| "переименуй везде" | rename across codebase | Предложить: grep сначала, потом точечные правки |
| "рефактор" без границ | "перепиши модуль" | Уточнить scope, разбить на ветки |
| "проверь все файлы" | lint всего проекта | Предложить: только изменённые файлы |
| Много связанных файлов | изменение интерфейса | Предложить: изолировать в одну ветку, работать по spec |
| Сессия уже длинная | 15+ файлов прочитано | Предупредить о приближении лимита |

---

## § 4 — Мониторинг сессии

Агент ДОЛЖЕН отслеживать приблизительное использование контекста в течение сессии:

**Ориентиры:**
- `/startgsession` — ~15 файлов прочитано → сессия уже нагружена
- Каждый большой файл (>100 строк) — ~1-2% бюджета
- Длинная беседа (20+ сообщений) — значительный контекст занят

**При ощущении приближения лимита** (субъективно, не точно):
> "⚠️ Мы уже прочитали много файлов в этой сессии. Контекст приближается к лимиту. Предлагаю: [завершить текущую задачу, закрыть сессию, продолжить через /continue]."

**Точный счётчик недоступен** — агент не может видеть оставшиеся токены напрямую. Оценка всегда приблизительная.

---

## § 5 — Принцип минимального чтения

| Ситуация | Читать | Не читать |
|---|---|---|
| Работа по harkly | handshake-harkly.md + current-context-harkly.md | global memory, regulations |
| Фикс бага в файле | только этот файл | весь модуль |
| Обновление контекста | только memory файлы | код проекта |
| Новая фича | spec.md + прямые зависимости | всё остальное |
| Ответить на вопрос | grep/glob для поиска | чтение всего проекта |

**Правило:** если можно ответить через grep/glob — никогда не читай файл целиком.

---

## § 6 — Компактные ответы

- Текстовые ответы — максимально краткие если не просят развёрнутого объяснения.
- Код — только изменённые части, не полный файл.
- Таблицы — только если реально структурируют информацию.
- Избегать повторения того что nopoint уже сказал в запросе.

---

## § 7 — Команда /continue

Если сессия прервалась из-за лимитов контекста — nopoint использует `/continue`.

Агент при `/continue` обязан:
1. Читать ТОЛЬКО хендшейк соответствующего проекта (1 файл).
2. Спросить: "На чём остановились?" — 1 вопрос.
3. Возобновить работу с минимальным чтением.
4. НЕ перечитывать регуляции, конституцию, semantic context.
