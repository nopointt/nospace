## Goal

Remote MCP Server на Cloudflare Worker. Claude Desktop, ChatGPT, Grok, Gemini, Cursor — подключаются через OAuth 2.1 и работают с knowledge base пользователя.

---

## Architecture — Three-Layer Single Worker

```
┌─────────────────────────────────────────────────┐
│  Layer 1: workers-oauth-provider                │
│  OAuth 2.1 Authorization Server                 │
│  /oauth/token  /oauth/register  /authorize      │
│  /.well-known/oauth-authorization-server        │
│  /.well-known/oauth-protected-resource          │
│  Storage: OAUTH_KV (KV namespace)               │
│  Tokens: opaque (NOT JWT), AES-GCM encrypted    │
└───────────────────┬─────────────────────────────┘
                    │
    ┌───────────────┴───────────────┐
    │                               │
    v                               v
┌──────────────────────┐  ┌────────────────────────┐
│ Layer 2: defaultHdlr │  │ Layer 3: apiHandler    │
│ better-auth (user)   │  │ mcp-ts-core (MCP)      │
│ /authorize → consent │  │ POST /mcp → tools      │
│ /api/auth/* → login  │  │ ctx.props.userId = tid  │
│ Storage: AUTH_DB     │  │ Storage: KB_DB          │
└──────────────────────┘  └────────────────────────┘
```

**Key design decisions:**
- workers-oauth-provider issues **opaque tokens** (not JWT) → trust validation, read `ctx.props.userId`
- better-auth и workers-oauth-provider **complementary** (identity vs token lifecycle)
- SessionStore **MUST be KV-backed** (in-memory Map from mcp-ts-core doesn't survive restarts)

## Phases

### Phase 1: Worker Scaffold + OAuth Provider (1 session)

**Goal:** CF Worker `harkly-mcp` отвечает на `/.well-known/*`, принимает `/oauth/register`, выдаёт токены.

| # | Task | Files | Notes |
|---|---|---|---|
| 1.1 | Create `workers/harkly-mcp/` directory with its own `wrangler.toml` | `wrangler.toml` | Separate worker, NOT part of SolidStart Pages. Bindings: OAUTH_KV, MCP_SESSION_KV, AUTH_DB, KB_DB, AUTH_KV, VECTORIZE_INDEX, AI |
| 1.2 | `npm init` + install deps | `package.json` | `@cloudflare/workers-oauth-provider`, `@cyanheads/mcp-ts-core`, `better-auth`, `better-auth-cloudflare`, `hono`, `zod` |
| 1.3 | Create `src/index.ts` — OAuthProvider entrypoint | `src/index.ts` | Import OAuthProvider from package, configure `apiRoute: ["/mcp"]`, token TTL, scopes: `knowledge:read`, `knowledge:write` |
| 1.4 | Configure KV token storage | `wrangler.toml` | `OAUTH_KV` binding — workers-oauth-provider uses this automatically |
| 1.5 | Dynamic client registration | — | Built-in to workers-oauth-provider at `/oauth/register` (RFC 7591). No custom code needed. |
| 1.6 | `/.well-known/oauth-authorization-server` | — | Auto-generated by workers-oauth-provider (RFC 8414) |
| 1.7 | `/.well-known/oauth-protected-resource` | — | From mcp-ts-core (RFC 9728). May need explicit route. |

### Phase 2: Consent UI + better-auth Integration (1 session)

**Goal:** `/authorize` показывает consent page, better-auth проверяет/создаёт сессию, `completeAuthorization()` привязывает userId.

| # | Task | Files | Notes |
|---|---|---|---|
| 2.1 | Create `defaultHandler` in OAuthProvider config | `src/handlers/default.ts` | Handles all non-API routes: `/authorize`, `/login`, `/api/auth/*` |
| 2.2 | `/authorize` flow: parse → check session → consent/login | `src/handlers/authorize.ts` | Use `env.OAUTH_PROVIDER.parseAuthRequest(request)` to get OAuth params. Check better-auth session via `auth.api.getSession({ headers })`. If no session → redirect to `/login?redirect=/authorize?...`. If session → show consent UI. |
| 2.3 | Consent UI page | `src/handlers/consent-ui.ts` | HTML page: "Harkly · {client_name} wants to access your knowledge base". Scopes listed. Approve/Deny buttons. POST to `/authorize/confirm`. Style with Harkly design tokens (warm palette, soft corners, light theme). |
| 2.4 | POST `/authorize/confirm` → completeAuthorization | `src/handlers/authorize.ts` | On approve: `completeAuthorization({ userId: session.user.id, scope: requestedScopes, props: { userId: session.user.id, email: session.user.email } })`. Redirect to client with auth code. On deny: redirect with `error=access_denied`. |
| 2.5 | Login/signup page | `src/handlers/login-ui.ts` | Minimal HTML form. POST to `/api/auth/sign-in/email` (better-auth). On success → redirect back to `/authorize`. |
| 2.6 | better-auth per-request instantiation | `src/lib/auth.ts` | Copy pattern from harkly-web: `getAuth(env.AUTH_DB)`. Use same AUTH_DB binding (shared D1). |
| 2.7 | Route `/api/auth/*` to better-auth handler | `src/handlers/default.ts` | If path starts with `/api/auth/` → `auth.handler(request)` |

**Verification:**
```bash
# Open in browser:
# https://harkly-mcp.{account}.workers.dev/authorize?client_id=test&response_type=code&code_challenge=abc&code_challenge_method=S256&scope=knowledge:read&redirect_uri=http://localhost
# Expected: redirects to login → login → consent page → approve → redirect to localhost with ?code=...

# Exchange code for token:
curl -X POST https://harkly-mcp.{account}.workers.dev/oauth/token \
  -d "grant_type=authorization_code&code={code}&client_id={client_id}&code_verifier={verifier}&redirect_uri=http://localhost"
# Expected: JSON with access_token, refresh_token
```

### Phase 3: MCP Server + KV Session Store (1 session)

**Goal:** `POST /mcp` принимает MCP requests, валидирует OAuth token, выполняет tools.

| # | Task | Files | Notes |
|---|---|---|---|
| 3.1 | Create `apiHandler` — mcp-ts-core entrypoint | `src/handlers/mcp.ts` | `createWorkerHandler()` from `@cyanheads/mcp-ts-core`. Pass tools array, server info. |
| 3.2 | KV-backed SessionStore | `src/lib/session-store.ts` | Replace in-memory Map. Key: `mcp:session:{sessionId}`, value: JSON, TTL from staleTimeoutMs. Use `MCP_SESSION_KV` binding. **CRITICAL — without this, sessions lost on Worker restart.** |
| 3.3 | Wire OAuth token → tenantId | `src/handlers/mcp.ts` | workers-oauth-provider passes `ctx.props.userId` to apiHandler. Map to `tenantId` for all D1 queries. Set `MCP_AUTH_MODE=none` in mcp-ts-core (trust OAuthProvider validation). |
| 3.4 | Streamable HTTP transport config | `src/handlers/mcp.ts` | POST /mcp, content-type application/json. MCP spec 2025-06-18 compliant. |

**Verification:**
```bash
# With valid access_token:
curl -X POST https://harkly-mcp.{account}.workers.dev/mcp \
  -H "Authorization: Bearer {token}" \
  -H "Content-Type: application/json" \
  -d '{"jsonrpc":"2.0","method":"tools/list","id":1}'
# Expected: JSON with 6 tools listed
```

### Phase 4: 6 MCP Tools (1-2 sessions)

**Goal:** Все 6 tools работают, multi-tenant, hybrid search.

| # | Tool | Input | Handler Logic | D1 Query |
|---|---|---|---|---|
| 4.1 | `search_knowledge` | query, kbId?, limit? | Hybrid search: FTS5 (BM25) + Vectorize (cosine) + RRF merge. Query rewrite (3-5 variants). | `chunks WHERE kb_id IN (user's KBs)` + Vectorize query |
| 4.2 | `get_document` | id | Fetch document + its chunks | `documents WHERE id = ? AND kb_id IN (user's KBs)` |
| 4.3 | `list_projects` | — | List user's knowledge bases | `projects WHERE tenant_id = ?` |
| 4.4 | `list_sources` | kbId | List sources in KB | `sources WHERE kb_id = ? AND kb_id belongs to user` |
| 4.5 | `get_schema` | kbId | Get schema fields for KB | `schema_fields WHERE kb_id = ?` |
| 4.6 | `list_extractions` | kbId, schemaId? | Get extracted structured data | `extractions WHERE kb_id = ? AND schema_id = ?` |

**Tool definition pattern (Zod):**
```typescript
tool('search_knowledge', {
  description: 'Semantic + full-text hybrid search across knowledge base',
  input: z.object({
    query: z.string().min(1).max(500),
    limit: z.number().int().min(1).max(50).default(10),
    kbId: z.string().optional(),
  }),
  auth: ['knowledge:read'],
  async handler(input, ctx) {
    const tenantId = ctx.props.userId;
    // ... hybrid search logic
  }
});
```

**Hybrid Search Implementation (steal from cloudflare-rag):**
1. FTS5 query: sanitize input → `SELECT rowid, rank FROM chunks_fts WHERE chunks_fts MATCH ? ORDER BY rank ASC LIMIT ?` (rank ASC, not DESC!)
2. Vectorize query: embed query → `VECTORIZE_INDEX.query(vector, { topK, filter: { tenantId } })`
3. RRF merge: `score = Σ 1/(k + rank_i)` where k=60
4. Return top N merged results with source metadata

**Gotchas:**
- FTS5 `rank` sort: **ASC** not DESC (lower = better match)
- Sanitize FTS5 queries: escape `"`, `*`, `(`, `)`, `OR`, `AND`, `NOT`
- Batch D1 `WHERE id IN (...)` to ≤100 IDs (SQLite 999 variable limit)
- Query rewrite: generate 3-5 query variants via Workers AI for better recall

**Verification:**
```bash
# Test each tool via MCP:
curl -X POST .../mcp -H "Authorization: Bearer {token}" \
  -d '{"jsonrpc":"2.0","method":"tools/call","params":{"name":"list_projects","arguments":{}},"id":1}'
# Expected: user's projects array

# Multi-tenant isolation test:
# User A's token → list_projects → only A's data
# User B's token → list_projects → only B's data
```

## Risks & Mitigations

| Risk | Impact | Mitigation |
|---|---|---|
| SessionStore in-memory (mcp-ts-core default) | Sessions lost on restart, broken multi-instance | Phase 3.2: KV-backed store (CRITICAL, do first) |
| KV eventual consistency (token revocation delay) | Brief window where revoked token still works | Acceptable for MVP; add explicit check on write ops |
| FTS5 special chars crash | Malformed queries crash D1 | Sanitize before MATCH (Phase 4.1) |
| D1 999-var limit | Large `IN (...)` queries fail | Batch to ≤100 IDs |
| Consent UI design mismatch | Looks foreign vs main app | Use Harkly design tokens (warm palette, light theme) |
| ChatGPT CIMD (Client ID Metadata Discovery) | Needs `global_fetch_strictly_public` flag | Add to wrangler.toml only if needed |
| Separate Worker ≠ same domain | CORS, cookie sharing issues | OAuth is token-based (Bearer), no cookies needed for MCP. Consent UI runs on Worker domain. |
