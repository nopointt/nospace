## Scenario 1: Concurrent Queries (ramp 5→10→20 VUs, 4.5 min)

| Metric | Value |
|---|---|
| Total requests | 628 |
| Success rate | 15.45% (97/628) |
| Error rate | 84.55% — all 429 (rate limit: 60 QPM/user) |
| **Successful query avg** | **13s** |
| **Successful query p95** | **23.88s** |
| **Successful query max** | **29.54s** |
| Empty answer rate | 0% (when answer arrives, always has content) |

**Root cause of errors:** Single API token shared across 20 VUs → 20× rate limit pressure.
This is expected behavior confirming rate limiting works correctly.

**Key finding:** Query latency under load (13s avg) is 2x the no-load latency (6s).
Bottleneck: Groq LLM generation (8B rewrite + 70B answer = 2 sequential API calls).

## Recommendations

1. **Query latency under load is the primary concern.** Groq free tier has TPM limits causing queue-up.
   Mitigation: LLM provider chain already in place (Groq → NIM → DeepInfra).
   Verify failover works under load.

2. **Multi-user load test** would give more realistic results. Current test uses 1 token for all VUs.

3. **Rate limits are correctly configured** and fail-fast (no queue-up of rate-limited requests).

4. **Non-query endpoints are fast** even under concurrent load (health: ~100ms, list: ~220ms).
