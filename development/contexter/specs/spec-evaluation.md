## F-013: RAGAS Phase 1 — Proxy metrics (zero LLM cost)

**Priority: P2**

### Current state

Confirmed: no metrics instrumentation exists anywhere in the codebase. The query route (`src/routes/query.ts`) returns answers with no quality signals recorded. There is no `eval_metrics` table in the schema. There is no `/metrics` route registered in `src/index.ts`.

### Implementation

#### 1. New file: `src/services/evaluation/proxy.ts`

This module computes all 10 proxy metrics from data already produced by the RAG cycle (no extra LLM calls, no extra API calls). It is called after `rag.query()` returns and does not block the response — logging is fire-and-forget.

**Metric definitions and computation:**

| Metric | Type | How to compute |
|---|---|---|
| `retrieval_score_mean` | `REAL` | `mean(sources.map(s => s.score))` — 0.0 if no sources |
| `retrieval_score_max` | `REAL` | `max(sources.map(s => s.score))` — 0.0 if no sources |
| `retrieval_score_spread` | `REAL` | `max - min` of source scores — 0.0 if <= 1 source |
| `chunks_retrieved_count` | `SMALLINT` | `sources.length` |
| `answer_length_tokens` | `SMALLINT` | `result.tokenUsage.llmCompletionTokens` (already computed) |
| `lexical_overlap_score` | `REAL` | ROUGE-L: longest-common-subsequence between query tokens and answer tokens, normalized. Formula: `lcs_length / max(query_tokens.length, answer_tokens.length)`. Tokenize by splitting on whitespace+punctuation (`/[\s\p{P}]+/u`), lowercase. Return 0.0 if either is empty. |
| `retrieval_latency_ms` | `INTEGER` | Caller measures: `Date.now()` before `rag.query()` (embedding + search phase only — see timing instrumentation below) |
| `generation_latency_ms` | `INTEGER` | Caller measures: time from end of retrieval to end of `generateAnswer()` |
| `empty_answer_rate` | `BOOLEAN` | `answer.trim().length === 0` |
| `embedding_l2_norm_mean` | `REAL` | Mean L2 norm of the query embedding vectors. `mean(queryEmbeddings.embeddings.map(e => sqrt(sum(e.vector.map(v => v*v)))))`. Computed from the `queryEmbeddings` returned by `embedBatch()` inside the RAG cycle. |

**Timing instrumentation** requires minor additions to `RagService.query()`:
- Return `retrievalLatencyMs` and `generationLatencyMs` in `RagAnswer` (extend the type).
- `retrievalLatencyMs`: time from `rewriteQuery` start to last `vectorStore.search()` completing.
- `generationLatencyMs`: time from `buildContext()` end to `generateAnswer()` returning.

**ROUGE-L function signature:**
```typescript

```

**Proxy metrics record type:**

#### 4. Retention cron

A BullMQ **separate queue** named `"maintenance"` runs a daily job at 02:00 UTC:
- Deletes raw `eval_metrics` rows older than 90 days
- Inserts one row per day into `eval_metrics_daily_agg` (see DDL below) with aggregated values before deleting

The maintenance worker is initialized in `src/index.ts` alongside the pipeline worker. Use a separate `Queue` instance named `"maintenance"` to avoid interfering with `"pipeline"` concurrency settings.

**Cron schedule:** add to queue on startup with `{ repeat: { cron: "0 2 * * *" } }`.

## F-031: RAGAS Phase 2 — Sampled LLM evaluation

**Priority: P3 | Gate: F-013 must be deployed and eval_metrics table confirmed present**

### Current state

`eval_metrics` table exists (from F-013). `llm_eval` JSONB column is present but NULL. No LLM evaluation infrastructure exists. `LlmService` supports arbitrary `chat()` calls. BullMQ queue infrastructure is established in `src/services/queue.ts`.

### Implementation

#### 1. Sampling decision

Performed in `src/routes/query.ts`, immediately after `rag.query()` returns (before fire-and-forget metrics log):

```typescript
const EVAL_SAMPLING_RATE = parseInt(process.env.EVAL_SAMPLING_RATE ?? "10", 10) // default 10%

// Deterministic hash so the same query_id always produces the same sampling decision
function shouldSample(queryId: string, rate: number): boolean {
  // FNV-1a 32-bit hash of queryId string, take mod 100
  let hash = 2166136261
  for (let i = 0; i < queryId.length; i++) {
    hash ^= queryId.charCodeAt(i)
    hash = (hash * 16777619) >>> 0
  }
  return (hash % 100) < rate
}

if (shouldSample(queryId, EVAL_SAMPLING_RATE)) {
  getLlmEvalQueue(process.env.REDIS_URL ?? "redis://localhost:6379")
    .add("llm-eval", { queryId, query: body.query.trim(), answer: result.answer, context })
    .catch((err) => console.error("llm-eval enqueue failed", err instanceof Error ? err.message : String(err)))
}
```

**Important:** `context` (the assembled context string from `buildContext()`) must be passed through `RagAnswer` or returned separately from `rag.query()`. Add `context: string` to `RagAnswer` type.

`EVAL_SAMPLING_RATE` env var: 0 disables LLM eval entirely, 100 evaluates every query. Default 10.

#### 2. New BullMQ queue: `"llm-eval"`

Separate from `"pipeline"` and `"maintenance"`. Worker concurrency: 1 (Groq rate limit awareness). Retry: `{ attempts: 2, backoff: { type: "exponential", delay: 30_000 } }`. `removeOnComplete: { age: 86_400 }`, `removeOnFail: { age: 7 * 86_400 }`.

#### 3. New file: `src/services/evaluation/llm-eval.ts`

Contains the BullMQ worker setup and the 3 Groq LLM calls.

**Job data type:**
```typescript
export interface LlmEvalJobData {
  queryId: string
  query: string
  answer: string
  context: string
}
```

**Worker processing — 3 sequential Groq calls:**

**Call 1 — Claim Extraction**

Model: `llama-3.1-8b-instant`. Max tokens: 512.

System prompt:
```
You are a precise claim extractor. Given an answer text, extract every distinct factual claim as a JSON array of strings. A claim is a single self-contained statement that can be verified as true or false independently. Output ONLY valid JSON. Example output: ["The sky is blue.", "Water boils at 100°C at sea level."]
```

User prompt:
```
Answer: {answer}

Extract all factual claims. Output only a JSON array of strings.
```

Parse response: `JSON.parse(response)` → `string[]`. On parse failure: `claims = []`, log warning.

**Call 2 — Verdict per claim**

Model: `llama-3.1-8b-instant`. Max tokens: 1024.

System prompt:
```
You are a precise fact-checker. Given a context and a list of claims, determine whether each claim is supported by the context. For each claim output "supported" or "unsupported". Output ONLY valid JSON as an array of objects with keys "claim" and "verdict". Example: [{"claim": "The sky is blue.", "verdict": "supported"}]
```

User prompt:
```
Context:
{context}

Claims:
{JSON.stringify(claims)}

For each claim, output whether it is "supported" or "unsupported" by the context. Output only a JSON array.
```

Parse response: `JSON.parse(response)` → `{ claim: string; verdict: "supported" | "unsupported" }[]`. On parse failure: treat all claims as unsupported.

Compute faithfulness: `supported_count / total_claims`. If `total_claims === 0`: faithfulness = `null`.

**Call 3 — Answer Relevancy**

Model: `llama-3.1-8b-instant`. Max tokens: 256.

System prompt:
```
You are an answer quality evaluator. Given a question and an answer, rate how relevant and complete the answer is to the question on a scale from 0.0 to 1.0. Output ONLY a JSON object with key "relevancy_score" and a float value between 0.0 and 1.0. Example: {"relevancy_score": 0.85}
```

User prompt:
```
Question: {query}

Answer: {answer}

Rate the relevancy and completeness of the answer to the question. Output only a JSON object.
```

Parse response: `JSON.parse(response)` → `{ relevancy_score: number }`. Clamp to `[0, 1]`. On parse failure: `relevancy_score = null`.

**JSONB payload stored in `eval_metrics.llm_eval`:**
```json
{
  "faithfulness": 0.85,
  "relevancy_score": 0.72,
  "claims_total": 7,
  "claims_supported": 6,
  "claims": [
    { "claim": "...", "verdict": "supported" },
    { "claim": "...", "verdict": "unsupported" }
  ],
  "evaluated_at": "2026-03-28T12:00:00Z",
  "model": "llama-3.1-8b-instant"
}
```

**Update query:** `UPDATE eval_metrics SET llm_eval = $1 WHERE query_id = $2`

If `query_id` row not found (race condition on startup): log warning, no-op.

#### 4. Cost estimate

At 10% sampling, 100 queries/day: ~10 eval runs/day × 3 calls × ~1500 tokens/call = ~45K tokens/day on Groq. At $0.05/M input tokens: ~$0.002/day, ~$0.07/month. At 1K queries/day: ~$0.70/month. Acceptable.

## F-032: RAGAS Phase 3 — Golden test set

**Priority: P3 | Gate: F-031 must be deployed; LLM eval infrastructure must be operational**

### Current state

`evaluation/golden/` directory does not exist. No CI evaluation workflow exists. `src/services/evaluation/llm-eval.ts` exists from F-031 and provides the 3-call evaluation logic that will be reused here. No RAGAS Python integration exists — this spec uses a TypeScript-native approach via direct Groq calls, consistent with the existing stack.

### Implementation

#### 1. Directory structure

```
evaluation/
  golden/
    README.md          -- format documentation
    manual/            -- hand-crafted Q&A pairs
      001.json
      002.json
      ...
    synthetic/         -- LLM-generated pairs, human-reviewed
      001.json
      002.json
      ...
    index.json         -- manifest: all pair IDs, source type, added date
```

#### 2. Q&A pair file format

Each JSON file in `manual/` or `synthetic/`:
```json
{
  "id": "gq-001",
  "question": "What document formats does Contexter support?",
  "expected_answer": "Contexter supports PDF, DOCX, TXT, and Markdown files.",
  "expected_sources": ["doc_abc123"],
  "tags": ["capabilities", "formats"],
  "added_by": "manual",
  "added_at": "2026-03-28",
  "reviewed": true
}
```

Fields:
- `id`: `gq-NNN` format (manual) or `gs-NNN` (synthetic)
- `question`: the test query string
- `expected_answer`: reference answer for relevancy comparison
- `expected_sources`: optional array of document IDs that should appear in sources (for retrieval recall)
- `tags`: free-form tags for filtering
- `added_by`: `"manual"` or `"synthetic"`
- `reviewed`: must be `true` for a pair to be included in CI runs

#### 3. Golden set generation script

New file: `evaluation/generate-synthetic.ts`

Uses `LlmService` with Groq to generate synthetic Q&A pairs from existing KB documents.

**Flow:**
1. Query PostgreSQL for up to 50 random chunks (stratified across documents)
2. For each chunk, call Groq with the generation prompt:

**System prompt:**
```
You are a test data generator for a retrieval-augmented generation system. Given a text chunk from a knowledge base, generate a realistic question that a user might ask whose answer is contained in this chunk. Also generate the ideal answer to that question based solely on the chunk content. Output only valid JSON.
```

**User prompt:**
```
Text chunk:
{chunk_content}

Generate a JSON object with keys "question" and "expected_answer". The question should be natural, specific, and answerable from this chunk. The answer should be accurate and complete based on the chunk.
```

**Output:** write to `evaluation/golden/synthetic/NNN.json`. Human must set `"reviewed": true` before pairs count for CI.

**Target:** 20 manual pairs + 30 synthetic pairs reviewed = 50 total for initial CI gate.

**Script invocation:**
```bash
bun run evaluation/generate-synthetic.ts --count 30 --output evaluation/golden/synthetic/
```

#### 4. CI evaluation runner

New file: `evaluation/run-eval.ts`

**Flow:**
1. Read all `*.json` from `evaluation/golden/manual/` and `evaluation/golden/synthetic/` where `reviewed === true`
2. For each pair, call the local API (`POST /api/query`) with the question
3. Run the 3 LLM eval calls from `src/services/evaluation/llm-eval.ts` (imported directly, not via queue)
4. Compute per-pair: faithfulness, relevancy, retrieval_recall (if `expected_sources` present)
5. Compute aggregate: mean faithfulness, mean relevancy, pass/fail per threshold

**Thresholds (CI gate — fail build if not met):**
- `mean_faithfulness >= 0.70`
- `mean_relevancy >= 0.65`

**Output format (stdout + JSON file):**
```json
{
  "run_at": "2026-03-28T12:00:00Z",
  "total_pairs": 50,
  "mean_faithfulness": 0.81,
  "mean_relevancy": 0.74,
  "pass": true,
  "failing_pairs": [],
  "per_pair": [...]
}
```

Write output to `evaluation/results/eval-{timestamp}.json`.

**Script invocation:**
```bash
bun run evaluation/run-eval.ts --api-url http://localhost:3000 --token $EVAL_TOKEN
```

Exit code 0 = pass, exit code 1 = fail (thresholds not met).

#### 5. CI workflow

New file: `.github/workflows/eval.yml` (or equivalent CI config if not GitHub Actions — check existing CI setup in repo root):


**Stale set management:** when new documents are added to a KB, existing pairs may become stale. Monthly review: run `evaluation/check-stale.ts` (simple script) to flag pairs where the API no longer retrieves `expected_sources`. Stale pairs: set `"reviewed": false` until re-verified.

No drift detection logic, no canary queries, no alerting infrastructure for RAG quality signals. Netdata is the existing monitoring stack on Hetzner (referenced in the implementation plan).

### Implementation

#### 1. New file: `src/services/evaluation/drift.ts`

**Embedding drift check — weekly BullMQ job**

**Flow:**
1. Fetch a random sample of 500 embeddings from `chunks` table for the requesting user (or global if run as system job): `SELECT embedding FROM chunks ORDER BY RANDOM() LIMIT 500`
2. PCA projection from 1024d → 32d (implementation below)
3. Compute MMD (Maximum Mean Discrepancy) between current week's sample and baseline sample
4. Store result in `eval_drift_checks` table (see DDL)
5. Alert if MMD > threshold

**PCA to 32d (no external library — pure TypeScript on Bun):**
- Compute mean vector across 500 samples (1024-dim mean subtraction)
- Build covariance estimate using 32 random projection vectors (random projection is acceptable approximation for drift detection — full SVD is too expensive on ARM 4GB)
- Random projection: generate 32 random unit vectors, project all 500 samples → 500×32 matrix
- This is a Johnson-Lindenstrauss sketch, not true PCA, but captures 90%+ of drift signal at 1/32 cost
- Store the 32-dim projections as the "current sample"

**MMD computation:**
- Gaussian kernel: `k(x,y) = exp(-||x-y||² / (2 * sigma²))` where `sigma = 1.0` (default bandwidth)
- MMD² = `mean_k(X,X) - 2*mean_k(X,Y) + mean_k(Y,Y)` where X = current 32d projections, Y = baseline
- Baseline: stored in `eval_drift_baseline` table (snapshot taken on first run, updated monthly)
- For 500 samples at 32d: O(500² × 32) — ~8M ops. Acceptable on ARM in <10s

**Threshold:** `MMD² > 0.05` triggers alert (configurable via `DRIFT_MMD_THRESHOLD` env var, default `0.05`).

**Weekly schedule:** BullMQ cron `"0 3 * * 1"` (Monday 03:00 UTC) in the `"maintenance"` queue (already created in F-013). Add this job type alongside the daily aggregate job.

#### 2. Canary queries on deployment

New script: `evaluation/canary.ts`

**Flow:**
1. Read first 10 reviewed pairs from golden test set (sorted by `added_at` ascending — oldest, most stable)
2. For each pair: call `POST /api/query`, compute retrieval_score_mean and answer length
3. Compare against 7-day rolling average from `eval_metrics` (proxy metrics)
4. Fail if any of:
   - Any individual query returns 0 sources (empty retrieval)
   - Mean retrieval_score_mean across 10 canary queries is < 70% of rolling baseline
   - More than 2 of 10 queries return empty answers

**Invocation (deployment hook):**
```bash
bun run evaluation/canary.ts --api-url $API_URL --token $EVAL_TOKEN
# Exit 0 = canary passed, exit 1 = canary failed → roll back deployment
```

Add to deploy script (or CI post-deploy step) as a blocking check before traffic is switched.

#### 3. Alert thresholds and signals

Two independent monitoring signals must BOTH fire to trigger an alert (multi-signal confirmation to avoid false positives):

| Signal | Threshold | Source |
|---|---|---|
| Faithfulness 7-day MA | < 0.70 | `eval_metrics.llm_eval->>'faithfulness'` rolling 7d mean |
| Retrieval score mean drop | > 10% drop from 30d baseline | `eval_metrics.retrieval_score_mean` |
| Embedding drift MMD | > 0.05 | `eval_drift_checks.mmd_score` |

Alert requires ≥ 2 of the above 3 signals firing simultaneously.

#### 4. Netdata alert configuration

File: `monitoring/netdata-ragas-alerts.conf`

Netdata can scrape custom metrics via its Python plugin or via a custom collector script. The approach here is a **custom shell collector** (simplest, no Python dependency):

New file: `monitoring/ragas-metrics-collector.sh` — called by Netdata every 60s:
```bash
#!/bin/bash
# Query PostgreSQL for RAG quality metrics, output in Netdata format
# netdata expects: SET variable_name value
FAITHFULNESS=$(psql "$DATABASE_URL" -At -c "
  SELECT COALESCE(AVG((llm_eval->>'faithfulness')::float), -1)
  FROM eval_metrics
  WHERE queried_at > NOW() - INTERVAL '7 days'
    AND llm_eval IS NOT NULL;")
echo "SET ragas_faithfulness_7d $FAITHFULNESS"

RETRIEVAL=$(psql "$DATABASE_URL" -At -c "
  SELECT COALESCE(AVG(retrieval_score_mean), -1)
  FROM eval_metrics
  WHERE queried_at > NOW() - INTERVAL '7 days';")
echo "SET ragas_retrieval_score_7d $RETRIEVAL"
```

Netdata alert rule in `monitoring/netdata-ragas-alerts.conf`:
```conf
alarm: ragas_faithfulness_low
     on: ragas.faithfulness_7d
  lookup: average -7d unaligned of ragas_faithfulness_7d
    warn: $this < 0.70
    crit: $this < 0.50
    info: RAG faithfulness 7-day average below threshold
      to: sysadmin

alarm: ragas_retrieval_drop
     on: ragas.retrieval_score_7d
  lookup: average -1d unaligned of ragas_retrieval_score_7d
    warn: $this < 0.015
    info: RAG retrieval score dropped significantly
      to: sysadmin
```
