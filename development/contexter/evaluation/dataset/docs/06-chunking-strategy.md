# Contexter Chunking Strategy

## Overview

Contexter uses a multi-strategy chunking pipeline that adapts to document structure. The system classifies each block of text and applies the optimal splitting strategy.

## Block Classification

The block classifier identifies five types:
- **code**: fenced code blocks, inline code, indented blocks
- **table**: markdown tables, pipe-delimited rows
- **list**: ordered and unordered lists with nesting
- **heading**: markdown headings (h1-h6)
- **text**: paragraphs, prose, everything else

Atomic blocks (code, table) are never split mid-structure. They are kept whole even if they exceed the target chunk size.

## Hierarchical Chunking

When a document section exceeds the maximum chunk size (default 512 tokens), the system creates a parent-child hierarchy:

1. The full section becomes a **parent chunk** (stored but not embedded individually)
2. The section is split into **child chunks** at natural boundaries (paragraphs, sentences)
3. Each child chunk is embedded independently
4. At query time, if multiple children from the same parent are retrieved, the parent content is used for richer context

Auto-merge threshold: 0.4 cosine similarity. Adjacent chunks that are very similar are merged to reduce redundancy.

## Contextual Prefixes

Each chunk receives a contextual prefix generated by Groq Llama 8B. The prefix is a 1-2 sentence summary of the document's topic and the chunk's position within it. This helps the embedding model produce better vectors by providing global context.

Example prefix: "This document describes Contexter's pricing model. This section covers the Pro tier features and limitations."

## Token Counting

BPE tokenizer (cl100k_base, GPT-3.5/4 vocabulary) is used for accurate token counting. The previous word-count heuristic had 8-69% error on code and CJK content.
