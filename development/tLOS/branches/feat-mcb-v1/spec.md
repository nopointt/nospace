## Задача в одном предложении

Реализовать **Marketing Command Board** — первое enterprise-приложение поверх tLOS, которое объединяет разрозненные аутстаф-маркетинговые команды в единую операционную систему с денежной моделью каждого действия.

---

## Контекст

**Клиент:** Артём, операционный менеджер proxy.market (ru.proxy.market)

**Боль:**
- 5 аутстаф-команд (SEO, Paid, SMM, Analytics, SERM) работают разрозненно
- Данные живут в Telegram, Excel, Google Sheets — хаотично, без единой схемы
- Владимир (фаундер) получает PDF-отчёты раз в 2-4 недели, не решения
- Никто не считает влияние на выручку — только трафик и позиции
- Петля обучения не замкнута ни разу: действия сделаны, атрибуции нет

**Источники:**
- `/nospace/docs/tLOS/MCB/Marketing Command Board (MCB).md` — техдок от Артёма (первичный)
- `/nospace/docs/tLOS/MCB/mcb_context.md` — полный семантический срез proxy.market

## Что строим (scope v1)

### BB-subset для MCB

Из полного BB-фреймворка в v1 нужны только 4 уровня:

```
ПОДВАЛ     — нормализация хаотичных входов (Telegram, Excel, API)
ОПЕРАЦИЯ   — 6 акторов-супервайзеров (по каналам)
ТАКТИКА    — эксперименты, разрывы, гипотезы, эскалация
СТРАТЕГИЯ  — цели, North Star, денежная модель

НЕ СТРОИМ в v1:
  Крыша    — исследование возможностей (следующий этап)
  Каркас   — регуляция самой системы (когда будут 3+ клиентов)
  Стены    — внутренний мониторинг
  Фасад    — кибербезопасность периметра
```

### 5 слоёв MCB (по Артёму) → BB-уровень

| Слой MCB | BB-уровень | Сущности |
|---|---|---|
| 1. Стратегия и цели | Стратегия | Цель, Состояние, Конвертер ценности |
| 2. Метрики | Операция + Подвал | Факт, Источник, Коннектор, Пайплайн |
| 3. Акторы | Операция | Задача, Действие, Actor-контракт |
| 4. Эксперименты | Тактика | Гипотеза, Эксперимент, Разрыв |
| 5. Блокеры | Тактика → Стратегия | Разрыв (критический), Эскалация |

## Архитектура tLOS

### 6 акторов (Wasm-компоненты с WIT-интерфейсами)

```
mcb-seo        — SEO/AEO: органика, позиции, нейровыдача
mcb-paid       — Paid Ads: Яндекс + Google кампании
mcb-smm        — SMM + Email: соцсети, email, B2B лиды
mcb-analytics  — Сквозная аналитика: атрибуция, UTM, аномалии
mcb-serm       — Репутация: отзывы, рейтинги, реакция
mcb-meta       — CMO AI: оркестратор, 3 сигнала наверх
```

### Когнитивный цикл (единый WIT-контракт всех акторов)

```
Observe    → собрать сигналы из назначенных источников
Diagnose   → выявить отклонение от цели (Разрыв)
Attribute  → связать отклонение с каналом / экспериментом
Decide     → сформировать рекомендацию
Quantify   → оценить влияние на выручку (₽/мес или Δ метрики)
Escalate   → если порог превышен — алерт в mcb-meta
```

Каждый актор реализует этот цикл. mcb-meta агрегирует выходы всех 5 и выдаёт:
- Где теряем деньги прямо сейчас
- Где масштабировать (ROI > X)
- Где риск (порог приближается)

### Слой данных (Подвал)

Проблема: входные данные хаотичны. Решение — единый Raw Layer:

```
Источник         Тип              Приоритет правды
─────────────────────────────────────────────────
Alytics          API/выгрузки     МАКСИМАЛЬНЫЙ (выручка, конверсии)
Админ-панель     API/БД           МАКСИМАЛЬНЫЙ (регистрации, оплаты)
Яндекс.Метрика   API              ВЫСОКИЙ (трафик, сессии)
Topvisor         API/выгрузки     ВЫСОКИЙ (позиции, видимость)
Telegram         Файл-экспорт     НИЗКИЙ (контекст, задачи)
Excel/Sheets     Файлы            НИЗКИЙ (ручной ввод)
```

Каждый входящий факт получает: `источник + время + уровень_доверия`
Без трёх атрибутов → не Факт, а Сигнал (не используется в решениях)

Статусы качества: `valid | warning | invalid | stale`

### Zero-Web2 constraint

tLOS запрещает HTTP/REST между внутренними акторами.
MCB-акторы общаются через **wRPC** (WIT over NATS).
Внешние интеграции (Topvisor, Alytics и др.) изолированы в Capability Providers.

- North Star дашборд для Владимира
- Конвертер ценности (если появятся другие продуктовые направления)

## Архитектурные задачи (вне MCB — уровень tLOS core)

### Web2 Gateway
Нужен системный документ `web2-gateway.md` — как tLOS консистентно взаимодействует с Web2.

**Три правила:**
1. Один вход — любой Web2 сигнал только через адаптер, никогда напрямую в актор
2. Нормализация на границе — адаптер конвертирует в tLOS-схему до пересечения периметра
3. Явная метка — факт Web2-происхождения несёт `origin: web2` навсегда

**Адаптеры для MCB v1:** Telegram, Метрика, Topvisor, Alytics, Admin API

**Связь с BB:** Web2 Gateway = Фасад (периметр системы)

**Статус:** не начато, нужен отдельный архитектурный документ

### TG бот для MCB
- Режим 1: пассивный — читает чат, Claude классифицирует сообщения → Сигналы в NATS
- Режим 2: активный — команды `/gap`, `/done`, `/block`, `/report`, `/goal`
- Архитектура: бот как Capability Provider на Фасаде, внутрь идут только нормализованные Сигналы
- `origin: web2 / trust: low` на всех данных из Telegram

**Статус:** не начато
