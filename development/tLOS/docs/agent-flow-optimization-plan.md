# Plan: Agent Flow Optimization
> Оптимизация потока задач в агентной иерархии tLOS
> Автор: nopoint + Assistant | 2026-03-13

---

## Зачем это нужно

Сейчас задача проходит через каждый уровень иерархии **дважды** — вниз (инструкции) и вверх (результат). Каждая остановка = вызов Claude = ~10-160 секунд. Полная цепочка: **5+ минут**, даже если всё прошло успешно.

**Цель:** успешный результат доходит до тебя за **~1 минуту** вместо 5. Ошибки решаются на месте если агент знает как, и поднимаются выше только если не знает или последствия серьёзные.

**Метрики успеха:**
- Скорость: время от намерения до результата
- Качество: разрыв между тем что ты хотел и тем что получил
- Стоимость: всё считаем в токенах

---

## Что меняется — 5 механизмов

### Механизм 1: Каскад инструкций (сверху вниз)

**Сейчас:** каждый уровень вызывает Claude чтобы "подумать" над задачей, потом передаёт дальше. Это 5 последовательных "подумать" = 5 ожиданий.

**Станет:** каждый уровень **обогащает и сужает** задачу для следующего. Не "думает с нуля", а добавляет свой контекст к тому что пришло сверху.

```
Ты: "Сделай кнопку logout"

Дирижёр добавляет:
  + Это задача для домена Development
  + Формализация: "UI элемент + обработчик выхода из сессии"
  → передаёт Chief/Development

Chief/Development добавляет:
  + Это работа для Lead/Frontend
  + Контракт: "кнопка в Header, рядом с Settings"
  → передаёт Lead/Frontend

Lead/Frontend добавляет:
  + Файл: Header.tsx
  + Использовать существующий Button component
  + Спека с acceptance criteria
  → передаёт Special

Special добавляет:
  + NATS subject для logout event
  + Проверочная команда
  → передаёт G3

G3 получает полностью готовую задачу → работает
```

**Что это даёт:** каждый уровень тратит минимум токенов. Не "придумывает с нуля" а "дополняет". Промпт для Claude на каждом уровне короче → ответ быстрее.

### Механизм 2: Быстрый путь успеха (снизу вверх)

**Сейчас:** G3 закончил → результат идёт к Special → Special к Lead → Lead к Chief → Chief к Дирижёру → тебе. Каждая передача = ожидание.

**Станет:**

```
G3 закончил успешно
  │
  ├──→ РЕЗУЛЬТАТ сразу летит к Дирижёру → тебе в Shell
  │    (прямой путь, ничего не блокирует)
  │
  └──→ УВЕДОМЛЕНИЕ параллельно всем промежуточным:
       Special: "задача X выполнена" → пишет LLM-summary в свою память
       Lead: "задача X выполнена" → записывает "ок" (без LLM)
       Chief: "задача X выполнена" → записывает "ок" (без LLM)
```

**Что это даёт:**
- Ты получаешь результат как только G3 закончил — не ждёшь 4 промежуточных уровня
- Special получает подробный summary (ему нужно для будущих спек)
- Lead и Chief получают факт "сделано" (им подробности не нужны — другой уровень абстракции)
- Стоимость уведомлений: ~5K токенов (1 LLM-вызов для Special summary) вместо ~200K (4 полных Claude вызова)

### Механизм 3: Матрица эскалации (при ошибках)

**Принцип:** не каждая проблема должна подниматься до генерала. Решение о том кто разбирается зависит от двух вещей:

1. **Blast radius** — насколько сильно решение влияет на систему
2. **Знакомость** — встречали ли мы этот баг раньше (проверяется по Bug Knowledge Base)

```
┌─────────────────────────────────────────────────────────────────┐
│                    КТО РЕШАЕТ                                   │
├────────────┬────────────────────────────────────────────────────┤
│            │                                                    │
│  SPECIAL   │  Blast radius: 1 файл                             │
│            │  Примеры:                                          │
│            │  - Заменить зависимость на эквивалентную           │
│            │  - Поправить баг внутри одного файла               │
│            │  - Retry после таймаута                            │
│            │  - Выбрать между двумя равноценными подходами      │
│            │                                                    │
├────────────┼────────────────────────────────────────────────────┤
│            │                                                    │
│  LEAD      │  Blast radius: несколько файлов одного поддомена   │
│            │  Примеры:                                          │
│            │  - Изменить интерфейс между файлами                │
│            │  - Добавить новый файл в поддомен                  │
│            │  - Пересмотреть спеку (Special не справился за 3)  │
│            │                                                    │
├────────────┼────────────────────────────────────────────────────┤
│            │                                                    │
│  CHIEF     │  Blast radius: весь домен                          │
│            │  Примеры:                                          │
│            │  - Новый сервис в Docker                           │
│            │  - Изменить схему таблицы в БД                     │
│            │  - Переназначить задачу другому Lead               │
│            │  - Изменить NATS subject                           │
│            │                                                    │
├────────────┼────────────────────────────────────────────────────┤
│            │                                                    │
│  ДИРИЖЁР   │  Blast radius: несколько доменов                   │
│            │  Примеры:                                          │
│            │  - Формат коммуникации между доменами              │
│            │  - Добавить новый домен                            │
│            │  - Бюджетные решения (модель, провайдер)           │
│            │                                                    │
├────────────┼────────────────────────────────────────────────────┤
│            │                                                    │
│  HUMAN     │  Blast radius: необратимо / стратегически          │
│  (ты)      │  Примеры:                                          │
│            │  - Удалить целый сервис                            │
│            │  - Сменить технологию                              │
│            │  - Деплой, ключи, доступы                          │
│            │  - Любое решение с PR в main                       │
│            │                                                    │
└────────────┴────────────────────────────────────────────────────┘
```

**Знакомость ошибки** проверяется по Bug Knowledge Base (новая pg таблица):
- Агент встретил ошибку → ищет похожую в базе
- Нашёл + есть рекомендованный путь решения → решает сам (если blast radius позволяет)
- Не нашёл → эскалирует на уровень выше
- После решения → записывает баг + решение в базу (для будущих случаев)

**Формула эскалации:**
```
Если (баг знакомый) И (blast radius ≤ мой уровень):
  → решаю сам, выбираю путь по: скорость + простота + стоимость (токены)

Если (баг знакомый) НО (blast radius > мой уровень):
  → решаю сам, НО уведомляю уровень выше

Если (баг незнакомый) И (blast radius ≤ мой уровень):
  → эскалирую на 1 уровень вверх

Если (баг незнакомый) И (blast radius > мой уровень):
  → эскалирую на 2+ уровня вверх
```

Границы полномочий каждого уровня — в YAML файле с диапазонами метрик. У каждого уровня есть пространство для самостоятельных решений в рамках своего blast radius.

### Механизм 4: Параллельный dispatch Chiefs

**Сейчас:** Дирижёр отправляет задачу Chief/Development, ждёт 160 секунд. Потом отправляет Chief/Design, ждёт ещё 130 секунд. Итого 290 секунд.

**Станет:** Дирижёр отправляет обоим одновременно. Оба работают параллельно. Итого ~160 секунд (время самого медленного).

```
Сейчас:                          Станет:
Chief/Dev ████████░░ 160s        Chief/Dev ████████░░ 160s
          Chief/Design ███████   Chief/Design ███████░░ 130s
          ─────────── 290s       ─────────────────── 160s
                                          (параллельно)
```

**Экономия:** ~45% времени на мульти-доменных задачах. 0 дополнительных токенов.

### Механизм 5: Кроссфункциональные задачи

Когда задача затрагивает несколько доменов и между ними есть зависимости:

```
Ты: "Сделай маркетинговый лендинг для новой фичи"

Дирижёр (через LLM) определяет:
  1. Какие домены нужны: Development, Design, Marketing
  2. Порядок: Development + Design (параллельно) → Marketing (ждёт обоих)
  3. Глобальная задача для каждого:
     - Chief/Dev: "API endpoint, контракт нужен Design и Marketing"
     - Chief/Design: "UI макет, API контракт получишь от Dev"
     - Chief/Marketing: "Копирайт, ждёшь макет + API"

Выполнение:
  Фаза 1: Chief/Dev + Chief/Design (параллельно)
  Фаза 2: Chief/Marketing (когда Фаза 1 готова)

Результат: агрегированный ответ от всех трёх
```

**Порядок определяет LLM** (не статические правила), потому что зависимости между доменами зависят от конкретной задачи. "Сделай лендинг" → Dev первый. "Обнови бренд" → Design первый.

## Новая служба: Debug Service ("Медики")

Третья мета-служба рядом с Samurizators и Regulators:

| Служба | Метафора | Что делает |
|---|---|---|
| **Samurizators** | Архивариусы | Компактят память (Operational → Global) |
| **Regulators** | Полиция | Следят за правилами (naming, RBAC, workflow) |
| **Debug** | Медики | Bug Knowledge Base: моделируют, проверяют, резолвят баги |

**Debug Service — три функции:**

1. **Modeling** — предсказывает баги ДО того как они случились. "Если поменять эту зависимость, вот что может сломаться"
2. **Verification** — проверяет реальные баги из логов агентов. "Этот баг действительно воспроизводится? В каких условиях?"
3. **Resolution** — пишет рекомендованные пути решения с impact analysis. "Вот 3 способа починить, вот стоимость каждого в токенах, вот что каждый может сломать"

**Bug Knowledge Base** (pg таблица):
- Каждый агент логирует встреченные ошибки
- Debug Service обогащает записи: impact analysis + рекомендованные пути
- При новой ошибке — агент сверяется с базой перед эскалацией

## Что делаем СЕЙЧАС vs ПОТОМ

### Сейчас (эта итерация)

| # | Что | Файлы | Результат |
|---|---|---|---|
| 1 | **Fast path** — результат G3 сразу к Дирижёру | `graph.py`, `bridge_handler.py` | Успех не ходит через 4 уровня |
| 2 | **Parallel Chiefs** — asyncio.gather вместо sequential | `graph.py` (dirizhyor_router_node) | 160s вместо 290s |
| 3 | **Escalation YAML** — границы полномочий + blast radius | `escalation_rules.yaml` (новый) | Каждый уровень знает что может решать сам |
| 4 | **Notification on success** — уведомления промежуточным | `graph.py`, `bridge_handler.py` | Special пишет summary, остальные пишут "ок" |
| 5 | **Cross-functional ordering** — LLM определяет порядок | `graph.py` (dirizhyor_node) | Дирижёр выдаёт execution_order + dependencies |
| 6 | **Bug KB table** — pg таблица для ошибок | `bug_kb.py` (новый) | Агенты пишут баги, читают решения |

### Потом (отдельные фазы)

| # | Что | Почему потом |
|---|---|---|
| A | **Горизонтальная коммуникация** (все уровни) | Большой этап: новый протокол + регламенты для каждого уровня |
| B | **Debug Service** (полная) | Нужна Bug KB сначала + моделирование = отдельный LLM pipeline |
| C | **Процедурно генерируемые регламенты** | Vision-уровень: регламенты как функция от параметров |
| D | **Авто-LLM скрипты** (без проверки Coach) | Нужна метрика error tolerance + статистика по задачам |

## Как проверим что работает

| Тест | Что проверяет | Ожидание |
|---|---|---|
| **Одиночный домен** | "Добавь logout кнопку" → только Chief/Dev | Результат за ~30-40s (вместо ~5 мин) |
| **Два домена параллельно** | "Добавь logout + обнови стиль" → Dev + Design | Время ≈ max(Dev, Design), не сумма |
| **Кроссфункциональная** | "Лендинг для фичи" → Dev+Design → Marketing | Фазовое выполнение, порядок корректный |
| **Знакомый баг** | Спровоцировать известную ошибку | Special решает сам, не эскалирует |
| **Незнакомый баг** | Спровоцировать неизвестную ошибку | Эскалация к Lead/Chief |
